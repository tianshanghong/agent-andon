# Agent Andon 托管版:零知识是怎么做到的(以及你怎么亲手验证)

> 这篇讲的是「托管版看板」(hosted relay)。本地自托管版本默认什么都不上传,不在本文范围内。
> 目标读者:想搞懂「为什么一个云服务器替我中转数据,却读不到我的内容」的人。

---

## 0. 一句话原理

**你的内容在离开你电脑之前就被一把只有你有的钥匙锁死了;中继服务器拿到的永远是一团它打不开的密文,它只负责"转发"和"在该提醒时提醒",从不"阅读"。**

更精确地说,有两层保证,强度不同,要分开讲:

1. **内容保密(强,密码学保证)**:中继即使被完全攻破,也拿不到你的项目名、消息、代码——这由 AES-256-GCM 加密保证,跟"信不信任运营者"无关。
2. **代码可信(中等,透明性保证)**:网页看板的代码是中继发给浏览器的,所以"它发的代码是不是老实的"不能由密码学单独保证,而是靠**可复现构建 + 公开透明日志**让"作弊"变得**会被公开抓到**。这一层是本文第 6、7 节的重点,也是你问的"验证原理"。

---

## 1. 谁能看到什么(信任模型)

我们**不信任**中继服务器(把运营者当成"好奇但不一定老实",甚至假设它被黑客攻破)。在这个前提下:

| 字段 | 中继看得到吗 | 说明 |
|---|---|---|
| 项目名 / 状态消息 / agent 名 / 战绩 | ❌ **看不到** | 全部加密,中继只有密文 |
| 会话 ID(sid) | ⚠️ 看到的是**哈希**,不是原文 | codex 的原始 sid 是项目路径,我们先 `sha256(boardId\|路径)` 再上传,中继拿不到路径 |
| 粗状态(忙/需要你/卡住/完成/空闲) | ✅ 看得到 | 用来决定"何时推送" |
| 时间、会话数、你的 IP、密文大小档位 | ✅ 看得到 | 不可避免的元数据(见第 8 节诚实边界) |

一句话:**中继知道"有没有事、什么时候、大概几个会话",但不知道"具体在做什么"。**

---

## 2. 核心机制:密封(seal)与开封(open)

整套系统的心脏就两个动作,都发生在**你这一侧**:

- **密封(seal)** —— 发生在你的电脑上(hook 里):
  把 `{项目名, 消息, agent, 战绩, 序号, 客户端时间}` 用钥匙 `K` 做 **AES-256-GCM** 加密,得到一团密文 `enc = {nonce, ct}`。然后把 `{哈希后的sid, 粗状态, 序号, enc}` 发给中继。
- **开封(open)** —— 发生在你的浏览器/手机上(看板里):
  从密文反解出原文,渲染成红绿灯卡片。

**钥匙 `K` 是 256 位随机数,在你电脑上生成,从不发给中继。** 中继全程只经手 `enc`(密文),没有 `K` 就永远打不开——这就是"零知识"的字面含义:中继对你的内容**零知识**。

```
你的电脑(hook)                     中继(云,不可信)              你的手机(看板)
  明文 ──seal(K)──▶ 密文 enc  ──────▶  存/转 enc(打不开) ──────▶  enc ──open(K)──▶ 明文
                                         ▲
                                         └── 它这里只有密文 + 粗路由,没有 K
```

---

## 3. 为什么中继"不能偷偷篡改"——三个细节

光"读不到"还不够。一个坏中继还可能想**骗你**(比如把"卡住"伪装成"一切正常")。我们用三招把它的手也绑住:

### 3.1 AAD 绑定:它不能张冠李戴,也不能在粗状态上撒谎
加密时,我们把**明文路由字段** `[boardId, sid, 状态, 序号]` 作为「附加认证数据(AAD)」一起绑进密文的完整性校验里。后果:
- 中继**不能**把 A 会话的密文挪到 B 会话/B 板子(AAD 对不上,开封直接失败)。
- 中继**不能**把它转发的"粗状态"改掉:如果它把密文标成"完成"、可密文其实是按"卡住"封的,**开封会失败**——你的看板会显示"无法解密/出错",而不是被骗成绿色。

> 换句话说:**那个唯一明文可见的"粗状态",中继也不能在它上面撒谎**,否则解密当场穿帮。

### 3.2 填充(padding):它看不出消息多长
密文按固定档位(256 / 1024 / 4096 字节…)填充后再加密,所以中继只能看出"这条消息落在哪个大小档",看不出精确长度——一句"完成"和一段长报错,在它眼里可能一样大。

### 3.3 单调序号(seq):它不能"重放旧状态"
每条事件带一个**严格递增**的序号。看板只接受比上次更新的序号。于是中继**不能**把一条旧的"一切正常"重新塞给你、盖住真实的"需要你"。
（**唯一做不到的**:中继可以**扣留/延迟**最新一条——这点无法靠任何中继消除,我们诚实写在边界里;但它**伪造不出**内容。）

## 4. 推送也保持零知识(最漂亮的一步)

手机推送有个天然难题:推送内容(比如「checkout-api 需要你」)得有人填,而中继读不到项目名,怎么填?

我们的做法是**双层包裹**:
1. 中继在该提醒时,把那团**密文 `enc`** 当作推送内容,经由系统推送服务(FCM/APNs)发出。
2. 手机上的 **service worker(后台脚本)从本地取出 `K`,把密文解开**,才得到「checkout-api 需要你」并弹出通知。

结果:**推送通知内容很丰富,但中继(以及苹果/谷歌的推送服务)全程只碰过密文。** 它只决定了"该响一声",没决定"响的内容写什么"。

---

## 5. 最难的一关:网页代码是中继发的(C1)

这是整个设计里最诚实、也最关键的一点,务必理解:

> AES-256-GCM 再强,也只保护"数据";可**运行解密的那段网页 JS 代码,是浏览器每次从中继下载的**。如果中继是恶意的(或被攻破),它完全可以多发一行 `把K偷偷发回来` 的代码,在你下次打开看板时拿走 `K`。

所以——**「即使我们被攻破也读不到你的内容」这句话,对一个"由中继自己发代码的网页版"是不成立的。** 这是密码学界对"网页版端到端加密"的经典警告(也是为什么浏览器版的加密邮件常被说成不可全信)。

我们不回避它,而是把承诺**诚实地分成三档**:

| 档位 | 承诺 | 要不要装东西 |
|---|---|---|
| **T1 纯网页** | "只要我们老实地发代码,就读不到你的内容" | 不用装 |
| **T2 网页 + 透明性(我们选的)** | "代码开源可复现、每个发出的版本都**公开记账**——我们**没法偷偷给你下黑手而不被抓到**" | 不用装(验证器可选) |
| **T3 安装版客户端** | "**即使被攻破也读不到**"(真·防攻破) | 要装(扩展/原生 app) |

我们把 **T2** 作为托管版的形态,**T3** 作为以后给"硬核隐私"用户的升级。下一节就是 T2 的"验证原理"。

---

## 6. 验证原理:你怎么确认中继没骗你

T2 的核心思想是把信任从**"相信他们读不到"**(做不到)换成**"他们没法偷偷动手脚而不被公开抓到"**(做得到)。靠三件套——这正是 WhatsApp Web / Signal 网页版在用的模型:

### ① 可复现构建(Reproducible Build)
看板代码完全开源,且**任何人都能从源码重新编译出**和中继所发**逐字节一致**的代码包。
（Agent Andon **零依赖**,这一点天然容易做到。）
→ 于是"中继发的代码 == 那份被审计过的开源代码吗?"变成一道**任何人都能算的哈希比对题**。

### ② 公开的、只能追加的透明日志(Transparency Log)
中继**每次**对外发布的代码包,它的 SHA-256 哈希都登记到一个**公开、只能追加、不能删改**的日志里(类比 HTTPS 证书界的 Certificate Transparency),并和某个 git 发布 tag 绑定。
→ 后果:中继**没法发布"没登记过"的代码**,也**没法只给某个目标用户单独发一份不一样的代码**——因为那个哈希要么不在公开日志里(被发现),要么在日志里但和开源源码对不上(被审出来)。

### ③ 可选的验证器(Verifier)
一个很小的浏览器扩展(就是 WhatsApp 的 "Code Verify" 那种),实时把**你浏览器真正收到的代码**做哈希,核对它"在不在日志里、和发布版一不一致",不一致就告警。

**这三件套买到的是:** 信任从"相信他们读不到"变成 **"他们要给你下黑手,就会留下公开可查的证据 / 当场被验证器拦下"**。对一个讲"诚实"的产品,这是一个**强且站得住**的承诺。

**诚实的局限(我们也写明)**:
- 不开验证器时,存在一个**时间窗(TOCTOU)**:理论上一次"定向投毒"可能在有人去审日志之前就把 `K` 偷走。要做到**实时阻止**,得用 T3(安装版)。
- 透明日志要**真有人去审**才有用(你、或社区)。它给的是"事后会被抓",不是"事前被物理阻止"。

## 7. 端到端的数据旅程(把上面串起来)

```
1. 你电脑上的 agent 状态变了(比如"需要审批")
        │  hook 触发
        ▼
2. seal:用 K 把 {项目名,消息,...} 加密成 enc;sid 先哈希
        │  POST /i/<board>(密文 + 粗状态 + 序号 + 写入令牌)
        ▼
3. 中继:验证写令牌 → 存下这团密文 → 看粗状态:从"忙"变"需要你"?
        ├──(实时)→ 通过 SSE 把这团密文广播给所有打开的看板
        └──(推送)→ 把这团密文经 FCM/APNs 发给你订阅过的手机
        ▼
4a. 看板(浏览器):收到密文 → 用 #fragment 里带来的 K → open → 渲染红绿灯卡片
4b. 手机 SW:收到密文 → 从本地取 K → open → 弹"checkout-api 需要你"
        ▲
        └── 第 3、4 步里,中继和推送服务全程只碰密文
```

钥匙怎么到看板:配对时 `K` 放在板子链接的 `#fragment` 里——**浏览器从不把 `#` 后面的内容发给服务器**,所以中继永远收不到 `K`(将来用扫码配对包装,你连复制都不用)。

## 8. 诚实边界:中继到底还能看到/做到什么

零知识 ≠ 隐形。把丑话说在前面(产品里也会这么写):

- **能看到的元数据**:你"在不在活动、大约什么时候(每条事件都有时间)、几个会话、你的 IP、密文大小档位"。这些是流量分析层面的,加密消除不了。
- **能做到的坏事**:一个恶意中继可以**延迟或扣留**某条提醒(让"需要你"晚到或不到)——但它**伪造不出**假内容,也**读不到**真内容。
- **读不到的**:prompt、代码、命令、项目名、消息、战绩。
- **本机假设**:`K` 存在你电脑上(`~/.andon`,权限 0600;以后用系统钥匙串)。如果**你自己的电脑**被入侵,那不在本设计的防护范围内。

一句产品话术(诚实版):**"连我们也读不到你的 prompt 和代码,只知道有没有事——而且我们发的代码是公开可复现、可验证的,想动手脚会被抓到。"**(注意:绝不在 T1/T2 上单独说"零知识"三个字而不加这层限定。)

---

## 9. 速记卡

- **读不到**:AES-256-GCM + 钥匙 K 永不离开你 → 密码学保证,与信任无关。
- **改不了**:AAD 绑定 → 张冠李戴/状态撒谎当场穿帮;序号 → 重放无效。
- **看不出长度**:大小档位填充。
- **推送也零知识**:发密文,手机 SW 本地解密。
- **唯一软肋(网页版)**:代码是中继发的 → 用**可复现构建 + 公开透明日志 + 可选验证器**把"作弊"变成"会被公开抓到";要"物理防攻破"就上 T3 安装版。
- **元数据照样泄露**:活动/时间/会话数/IP——诚实写明,不假装隐形。
